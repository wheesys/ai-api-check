## 1. 项目背景与目标

本项目是一个**中转站（API 代理）模型质量检测平台**（来源：项目 `CLAUDE.md`）。

**核心需求**
- 支持多协议中转站：OpenAI、Anthropic、**Gemini（Google Generative AI）** 三种上游协议，单个中转站可同时支持任意协议组合（如"兼容站"可同时勾选 OpenAI+Anthropic+Gemini）。
- Gemini 支持两种暴露形态：**原生协议**（`generativelanguage.googleapis.com`，`:generateContent`，`x-goog-api-key` 鉴权）与中转站用 **OpenAI 兼容层**（`/v1/chat/completions`，`model=gemini-*`）转出。
- 中转站可自定义**请求地址、API Key、名称**。
- 自动从中转站拉取支持的模型列表；拉取失败时支持手动输入。
- 输出各中转站不同模型的**质量报告**，可导出 **PDF**。
- **可视化**配置中转站与发起质量检测。

**核心价值定位（用户确认）**
> 平台最核心的价值是**模型来源真实性检测**——识别中转站是否"套壳换底"或"从 Codex 等订阅工具逆向转出 API 冒充官方直供"。其余维度（连通性、计费、能力）为辅助。

## 2. 需求澄清结论（决策记录）

| # | 决策点 | 选择 | 关键理由 |
|---|---|---|---|
| 1 | 部署形态 | **本地 Web 应用**（浏览器界面 + 本地存储 + Key 仅本地） | 部署简单、契合可视化需求、Key 不出本机更安全 |
| 2 | 检测维度 | 连通性与性能 / 计费一致性 / 能力探测 / **真实性检测（核心）** | 移除"输出正确性（题集评测）"，省去最大工作量与 token 消耗（YAGNI） |
| 3 | 真实性对照基线 | **无官方 key，纯启发式检测** | 无官方直连作对照，只能靠行为指纹/特征加权给"可疑度" |
| 4 | 真实性子目标 | a. 套壳/换底  b. **逆向/工具转出**（Codex / ChatGPT / Claude 订阅 / Cursor / Copilot / **Gemini CLI / AI Studio** 等逆向成 API） | 识别"非官方直供"来源 |
| 5 | 后端技术栈 | **Python（FastAPI 异步）** | tokenizer、异步探测、PDF 生态最成熟 |
| 6 | 前端技术栈 | **Vue 3 + Naive UI** | 配置型后台界面友好、中文社区活跃 |
| 7 | 数据存储 | **SQLite** | 本地轻量；遵循全局 DB 规则（自增主键/无外键/datetime/全注释） |
| 8 | **支持协议** | **OpenAI / Anthropic / Gemini 三协议** | 覆盖主流上游；适配器模式新增协议不改现有代码（SOLID-O） |
| 9 | **协议建模** | **协议集合多选**（站点存 `protocols` 集合，模型归属单一 `protocol`） | 取代固定的三/四类枚举组合，避免枚举组合爆炸，未来加协议零改枚举 |
| 10 | **Gemini 接入形态** | **原生协议（Developer 风格 + Vertex 风格两种端点）+ OpenAI 兼容层** 均支持 | 覆盖 AI Studio 直供、Vertex 原生路径透传、generativelanguage 风格封装、兼容层转出全部现状 |
| 11 | **Gemini 真实性主手段** | **功能性指纹探针：主动测能否跑通 Gemini 独有功能以鉴别"是不是真 Gemini"** | 真 Gemini 必能返回独有功能的特有结构化字段，套壳模型给不出；功能可用性比被动指纹更难伪造。上游 Studio/Vertex 仅附带信息，非判定重点 |

**术语约定**
- **TTFT（Time To First Token）**：流式请求中，从发出请求到收到第一个 token 的耗时，衡量"首字延迟"。非流式用"总耗时"。
- **套壳/换底**：中转站声称提供贵模型，实际转发给便宜/其他模型。
- **逆向/工具转出**：将 C 端订阅工具（Codex、ChatGPT Plus、Claude Pro、Cursor、Copilot、**Gemini CLI / Google AI Studio / Antigravity** 等）的额度逆向封装成兼容 API 出售，冒充官方直供。
- **协议（protocol）**：中转站上游遵循的 API 规范，取值 `openai`·`anthropic`·`gemini`。一个站可支持多协议；一个模型归属单一协议。
- **Gemini 原生协议**：Google Generative AI 的 REST 规范，端点 `:generateContent` / `:streamGenerateContent`，鉴权 `x-goog-api-key`，usage 字段为 `usageMetadata`。

## 3. 可行性分析

| 维度 | 可行性 | 实现路径 | 局限 |
|---|---|---|---|
| 连通性与性能 | ✅ 成熟 | HTTP 请求 + 计时 + 错误码统计 | 无 |
| 计费一致性 | ✅ 可行 | 申报单价 × usage + 本地 tokenizer 估算对比 + 官方基线对比（如有） | 看不到真实账单扣费；Anthropic 流式默认不返回 usage，需本地估算兜底 |
| 能力探测 | ✅ 可行 | 标准探针：上下文长度/流式/函数调用/多模态等 | 上下文长度探测较耗时（需递增输入） |
| **Gemini 协议适配** | ✅ 可行 | 原生 `:generateContent`/`:streamGenerateContent` + `usageMetadata`；`:countTokens` 精确计数；`GET /v1beta/models` 拉取模型；兼容层走 `/v1/chat/completions` | 兼容层下 Gemini 特有字段（safetyRatings/thoughtsTokenCount）可能被中转站抹平，真实性特征减弱 |
| 真实性检测（套壳+逆向转出） | ⚠️ 启发式可行 | 探针提取行为特征 + 加权评分 → 可信度分级 | **无法 100% 确认**；中转站可刻意抹除特征；仅给"高度可疑/可能/正常"分级 |
| PDF 报告 | ✅ 可行 | 前端 ECharts 渲染图表 + 后端 weasyprint 由 HTML 模板生成 | 无 |
| 本地部署 | ✅ 可行 | FastAPI + Vue 静态资源 + SQLite | 无 |

**结论**：项目整体技术可行。唯一需在产品层面诚实传达的是——**真实性检测是"可信度评分"而非"铁证"**。

```
                   └────────────────┘
```

**分层职责**
- **前端**：配置管理、任务发起与实时进度（SSE）、报告可视化与 PDF 触发。
- **后端 API 层**：CRUD、任务调度、SSE 推送。
- **检测引擎**：异步任务池执行探针序列，落库 `probe_record`。
- **Provider 适配器**：三类协议（OpenAI / Anthropic / Gemini）+ 兼容层统一为同一调用接口（适配器模式）。Gemini 适配器内部再分"原生协议"与"OpenAI 兼容层转出"两条路径。
- **真实性评分**：聚合探针特征 → 加权评分。
- **PDF 生成**：weasyprint 由 HTML 模板渲染。
- **数据层**：SQLite。

## 5. 关键技术决策

| 决策点 | 方案 | 理由 |
|---|---|---|
| 检测执行 | 异步任务池 + SSE 实时进度 | 单模型多探针耗时数十秒~分钟，阻塞会 HTTP 超时 |
| Provider 适配 | 适配器模式，OpenAI/Anthropic/Gemini 三协议统一接口 | 符合 SOLID-O，新增协议不改现有代码 |
| Token 计数 | tiktoken（OpenAI 兼容）+ 近似估算兜底（Anthropic）+ **Gemini `:countTokens` 端点 / 估算兜底** | 校验 usage 真实性 |
| PDF 导出 | 前端 ECharts 渲染 + 后端 weasyprint 由 HTML 生成 | 图表丰富、样式可控 |
| Key 存储 | 本地加密存储 + 日志脱敏 | 遵循安全规则（Key 不外发、不打印） |
| 数据层 | SQLite（自增主键/无外键/datetime/全字段注释） | 遵循全局 DB 规则 |
| 交付 | 开发式启动为主，可选 docker-compose（届时确认时区） | 本地自用最简 |

## 6. 数据模型设计（草案，**待确认**）

遵循全局规则 3.2：整数自增主键、无外键约束（业务字段关联）、`datetime` 类型、全字段带 COMMENT。

**① `relay_station`（中转站）**：id / name / protocols（**协议集合**，JSON 文本，元素 ∈ `openai`·`anthropic`·`gemini`；取代原单值 `station_type`，"兼容站"即多元素集合） / base_url / api_key(加密) / status / created_at / updated_at

**② `model`（模型）**：id / station_id(业务关联) / protocol（**该模型归属协议**，`openai`·`anthropic`·`gemini`，须 ∈ 所属站的 protocols） / access_mode（**接入形态**，`native` 原生协议·`openai_compat` 兼容层转出；非 Gemini 默认 `native`） / gemini_endpoint_style（**Gemini 原生端点形态**，`gemini_developer`·`vertex`，默认 `gemini_developer`；仅 gemini+native 有意义） / gemini_vertex_json（**Vertex 风格专属配置**，JSON 文本：`{project, location, auth_style}`；非 Vertex 留空，下放至模型级以支持同站不同模型属不同 project/location） / model_name / display_name / source(`fetched`·`manual`) / input_price(**TEXT 精确小数**，应用层 Decimal 计算) / output_price(**TEXT 精确小数**) / declared_context_length / enabled / created_at / updated_at

**③ `detection_task`（检测任务）**：id / station_id / model_id / status(`pending`·`running`·`completed`·`failed`·`canceled`) / progress / config_json(检测配置快照) / started_at / finished_at / created_at / updated_at / error_message

**④ `detection_result`（结果汇总）**：id / task_id / model_id / overall_score / connectivity_score / performance_score / billing_score / capability_score / authenticity_score / authenticity_subscores_json(套壳分+逆向转出分) / details_json / created_at

**⑤ `strategy_result`（策略检测结果）**：id / task_id / model_id(业务关联) / strategy_category（策略大类：`connectivity`·`performance`·`billing`·`capability`·`authenticity`） / strategy_key（策略细项标识，如 `ttft`·`throughput`·`billing_consistency`·`cap_function_call`·`gemini_thinking`·`gemini_code_execution`·`gemini_url_context`·`gemini_vertex_rag` 等） / strategy_name(策略中文名) / result_status（`pass`·`fail`·`degraded`·`skipped`；功能探针 supported/degraded/unsupported 映射于此） / score(该策略得分，可空) / weight(该策略在所属维度的权重快照) / metrics_json(量化指标，如 ttft_ms/命中的特有字段名) / evidence_json(判定证据，脱敏：命中/缺失字段、原始片段引用) / created_at
> **每个模型 × 每个检测策略一行**，独立保留每项策略的结论与证据，支撑可溯源的真伪/质量报告。

**⑥ `probe_record`（探针原始记录）**：id / task_id / strategy_result_id(业务关联，归属哪条策略结果) / probe_type / request_at / response_at / ttft_ms / http_status / success / usage_json / estimated_tokens / feature_flags_json / raw_response_json(脱敏) / error_message / created_at

**设计要点**
- 探针原始记录与结果汇总分离：支持溯源审计，评分逻辑可独立演进。
- 价格/上下文等可变信息快照进 `config_json`，保证历史报告可复现。
- **协议集合多选**：`relay_station.protocols` 存协议集合而非固定类型枚举，新增协议（未来如 Cohere/Bedrock）无需改枚举，仅扩集合取值与适配器（SOLID-O）。SQLite 无原生数组，用 JSON 文本字段承载，应用层解析。
- **模型归属单一协议**：每个 `model` 通过 `protocol` 明确归属，`access_mode` 区分 Gemini 走原生协议还是兼容层，决定探针调用哪条适配路径与如何解析 usage。
- **三层结果模型**：`detection_result`(模型综合分) → `strategy_result`(每模型每策略结论，逐项保留) → `probe_record`(策略下每次请求原始记录)，自顶向下可逐层下钻溯源。
- **价格精确存储**：SQLite 无原生 DECIMAL（NUMERIC 亲和性下小数退化为浮点），故 `input_price`/`output_price` 用 TEXT 存精确小数，应用层 Decimal 计算；若改用 PostgreSQL 则可直接用 NUMERIC。
- **Gemini 端点配置下放模型级**：`gemini_endpoint_style` 与 `gemini_vertex_json` 置于 `model`，支持同一中转站下不同 Gemini 模型属不同 project/location 与端点形态。

## 7. Gemini 协议适配要点

Gemini（Google Generative AI）的 API 规范与 OpenAI/Anthropic 差异较大，且中转站存在两种暴露形态，适配器内部据此分两条路径。

### 7.1 两条接入路径

| 路径 | `access_mode` / `endpoint_style` | 端点与鉴权 | usage 字段 | 适用场景 |
|---|---|---|---|---|
| Gemini 原生 · Developer 风格 | `native` / `gemini_developer` | `POST {base}/v1beta/models/{model}:generateContent`；流式 `:streamGenerateContent?alt=sse`；鉴权 `x-goog-api-key: <KEY>`（或 `?key=`） | `usageMetadata.{promptTokenCount, candidatesTokenCount, totalTokenCount, thoughtsTokenCount}` | AI Studio 直供站、以 generativelanguage 风格封装上游（含封装后的 Vertex）的中转站 |
| Gemini 原生 · Vertex 风格 | `native` / `vertex` | `POST {base}/v1/projects/{project}/locations/{location}/publishers/google/models/{model}:generateContent`；流式 `:streamGenerateContent`；鉴权 `Authorization: Bearer <token>`（或站点自定义） | 同上 `usageMetadata` | 中转站以 Vertex 原生路径透传（需配 project/location） |
| OpenAI 兼容层转出 | `openai_compat` / — | `POST {base}/v1/chat/completions`，`model="gemini-*"` | OpenAI 风格 `usage.{prompt_tokens, completion_tokens, total_tokens}` | 多数综合中转站把 Gemini 并入兼容层 |

### 7.2 请求/响应结构差异（原生路径）

- **请求体**：`{"contents":[{"role":"user","parts":[{"text":"..."}]}], "generationConfig":{...}, "systemInstruction":{...}}`；角色仅 `user`/`model`，无 `system` role（用 `systemInstruction`）。
- **流式**：`:streamGenerateContent?alt=sse` 返回 SSE，每帧为完整 `GenerateContentResponse` 增量；TTFT 取首个含 `candidates[].content.parts[].text` 的帧。
- **能力探测**：函数调用走 `tools[].functionDeclarations`；多模态走 `parts[].inlineData`/`fileData`；JSON 模式走 `generationConfig.responseMimeType="application/json"`；思考模型（gemini-2.5 系）含 `thinkingConfig` 与 `thoughtsTokenCount`。
- **安全字段**：响应含 `candidates[].safetyRatings`、`promptFeedback.blockReason`，为 Gemini 特有指纹。

### 7.3 模型列表与 token 计数

- **模型列表**：Developer 风格 `GET {base}/v1beta/models`（返回 `models[].name`、`supportedGenerationMethods`、`inputTokenLimit`/`outputTokenLimit`）；Vertex 风格 `GET {base}/v1/projects/{project}/locations/{location}/publishers/google/models`；兼容层走 `GET {base}/v1/models`。拉取失败回退手动输入（与现有规则一致）。
- **token 计数**：Developer 风格 `POST {base}/v1beta/models/{model}:countTokens`、Vertex 风格 `.../models/{model}:countTokens` 可精确计数，用于校验 `usageMetadata` 真实性；兼容层无对应端点，用估算兜底。

### 7.4 真实性检测 — Gemini 逆向/套壳特征

延续"逆向/工具转出"子目标，Gemini 的非官方直供来源指纹：

| 来源 | 特征指纹 | 加权方向 |
|---|---|---|
| **Google AI Studio 逆向** | 免费档模型可用但缺企业级配额特征；`safetyRatings` 结构异常或被裁剪；限流模式贴近 AI Studio 免费额度 | 可疑↑ |
| **Gemini CLI / Code Assist 逆向** | 注入特定 system 指令/工具壳痕迹；`modelVersion` 回显缺失或与官方不符；响应头缺 Google 直供特征 | 可疑↑ |
| **Antigravity / IDE 工具转出** | agentic 工具特有的会话/上下文包装残留；多模态/函数调用能力被阉割 | 可疑↑ |
| **套壳换底（声称 Gemini 实转他模型）** | `usageMetadata` 缺失或字段不全（无 `thoughtsTokenCount` 但声称 2.5 思考模型）；不支持 Gemini 特有 `responseMimeType`/`safetyRatings`；tokenizer 切分特征与 Gemini 不符 | 高度可疑↑ |
| **官方直供（正常基线）** | `usageMetadata` 字段齐全且与 `:countTokens` 一致；`safetyRatings`/`modelVersion` 完整规范；能力探针全通过 | 正常 |

> 同其余真实性维度：**仅给可信度分级（高度可疑/可能/正常），非铁证**；阈值可调，支持人工复核。兼容层转出会抹平部分原生特征，此时真实性判定置信度自动下调并在报告标注。

### 7.5 功能性指纹探针（主动探测，Gemini 真实性核心手段）

**核心思想**：**真正的 Gemini 模型必然能正确使用 Gemini 独有功能**，而套壳成其他模型（GPT/Claude/开源模型）的中转站给不出这些功能特有的结构化响应。因此主动调用 Studio / Vertex 的独有功能，校验响应是否返回该功能**特有的结构化字段**——**能正确返回即证明是真 Gemini**；报错 / 忽略参数返回普通文本 / 特有字段缺失 = 并非真 Gemini（套壳换底）。功能可用性是判定"是不是真 Gemini"的核心试金石，远比被动指纹可靠。

> **本节目标是真伪鉴别（是不是真 Gemini），不是上游溯源**。能跑通 Gemini 功能的就是真 Gemini，至于功能由 Studio 还是 Vertex 提供，仅作附带信息，不作为判定重点。

**判定三态**：每个探针输出 `supported`（返回特有字段且语义正确）/ `degraded`（部分返回或字段不全）/ `unsupported`（报错或完全忽略）。

#### A. 通用 Gemini 独有功能探针（真伪主判据，双向加权）

两个平台的真 Gemini 都应支持（视模型版本），套壳成非 Gemini 模型则给不出特有字段。**支持→证真、声称支持却不支持→证伪**。

| 探针 | 请求要点 | 真 Gemini 特有字段（须出现） | 套壳模型表现 |
|---|---|---|---|
| **思考（Thinking）** | 2.5/3 思考模型设 `thinkingConfig`（`thinking_level` 或 `thinking_budget`） | `usageMetadata.thoughtsTokenCount>0`；多轮回传 `thoughtSignature` | 声称思考模型但 thoughtsTokenCount 缺失/为 0；无签名 |
| **代码执行** | `tools:[{code_execution:{}}]` + 要求"执行代码得结果" | `parts[].executableCode.{language,code}` + `parts[].codeExecutionResult.{outcome,output}` | 只回代码文本、无执行结果字段 |
| **Google 搜索接地** | `tools:[{google_search:{}}]`（旧模型 `google_search_retrieval`）+ 问需实时信息的问题 | `groundingMetadata.{webSearchQueries, groundingChunks[].web.{uri,title}, groundingSupports, searchEntryPoint}` | 无 groundingMetadata / 报错 / 纯文本臆造 |
| **受控结构化输出** | `generationConfig.responseMimeType="application/json"` + `responseSchema`（含 enum / propertyOrdering） | 严格遵循 schema，字段顺序符合 propertyOrdering | 忽略 schema / 返回非结构化文本 |
| **上下文缓存** | `cachedContents.create` 建缓存后带 `cachedContent` 引用请求 | `usageMetadata.cachedContentTokenCount>0` + 计费折扣 | 缓存端点不存在 / 字段缺失 |
| **logprobs** | `generationConfig.responseLogprobs=true` + `logprobs=N` | `candidates[].logprobsResult.{topCandidates, chosenCandidates}` + `avgLogprobs` | 字段缺失 |
| **安全评级** | 普通请求观察 | `candidates[].safetyRatings[].{category,probability}`、`promptFeedback.blockReason` 结构规范 | 缺失 / 结构与 Gemini 不符 |
| **token 计数一致性** | `:countTokens` 与实际 `usageMetadata.promptTokenCount` 对比 | 两者一致（Gemini 自有 tokenizer 切分） | 偏差大 / 端点不存在 |
| **多模态时间戳** | 传视频 `fileData` + 问"第 N 秒发生什么" | 准确定位时间戳内容 | 不支持多模态 / 答非所问 |

#### B. 平台特有功能探针（单向确证，支持=铁证真 Gemini，不支持不扣分）

仅某一平台提供，受接入方式限制。**支持即一票确证为真 Gemini；不支持不能据此判套壳**（可能只是另一平台直供或模型版本不含该功能）。

| 探针 | 平台 | 仅此平台支持的特征 | 用途 |
|---|---|---|---|
| **URL Context** | 仅 Developer API（Studio） | `url_context_metadata.urlMetadata[].{retrievedUrl, urlRetrievalStatus}` 且内容反映该页 | 通过即证真 Gemini |
| **Vertex RAG 检索接地** | 仅 Vertex | 接受 `tools:[{retrieval:{vertexRagStore:{...}}}]` 并返回检索结果 | 通过即证真 Gemini |
| **Google Maps 接地** | 仅 Vertex（Gemini 3 企业） | 返回 Maps 接地元数据 | 通过即证真 Gemini |
| **SafetySetting.method** | 仅 Vertex | 接受 `safetySettings[].method=PROBABILITY\|SEVERITY` 并生效 | 通过即证真 Gemini |

#### C. 真伪判定逻辑（是不是真 Gemini）

- **A 组任一探针 supported** → 该功能特有结构化字段非 Gemini 模型无法伪造，**强力证真**；多个 supported → 基本确认真 Gemini。
- **B 组任一探针 supported** → **一票确证真 Gemini**（平台特有功能别的模型绝无法返回对应结构）。
- **声称为 Gemini 模型，但 A 组功能（在该模型版本应支持的范围内）大面积 unsupported** → **疑似套壳成非 Gemini 模型**，真实性可疑度↑。
- **A、B 全部 unsupported 且仅基础对话可用** → 高度可疑（套壳，或仅透传基础 generateContent 的残缺逆向）。
- 上游来自 Studio 还是 Vertex 由 B 组附带得出，**仅作报告备注，不影响真伪结论**。

#### D. 探针执行约束

- **协议路径限制**：功能性探针**仅对 `access_mode=native` 原生路径有效**；OpenAI 兼容层（`/v1/chat/completions`）通常不透传这些 Gemini 专属工具，兼容层下偏门功能 unsupported 是**预期而非证据**，此时降级为弱信号并在报告标注"兼容层无法做功能性指纹检测"。
- **成本与开关**：偏门功能探针（尤其搜索接地、代码执行、视频）消耗较高，须可单独勾选、设预算上限、支持采样而非全跑（YAGNI / 风险节 token 消耗）。
- **误报控制**：单一探针 unsupported 不直接判套壳（可能模型版本不支持该功能），需结合模型声明能力（`supportedGenerationMethods`）与多探针综合加权。

## 9. 风险与局限

1. **真实性检测非铁证**：启发式评分，中转站可对抗。报告须明示"可信度"语义，避免向用户承诺"100% 识别"。
2. **Anthropic usage 兜底**：流式默认不返回 usage，计费一致性需本地估算，精度受限。
3. **上下文长度探测耗时**：递增输入策略需设上限，避免长时间占用与高额消耗。
4. **token 消耗**：检测本身消耗中转站额度，需提供并发数/重复次数/预算上限配置。
5. **逆向探针误报**：行为特征可能误判，阈值需可调并支持人工复核。
6. **Gemini 兼容层特征丢失**：经 OpenAI 兼容层转出的 Gemini 会抹平 `safetyRatings`/`usageMetadata`/`thoughtsTokenCount` 等原生指纹，真实性与计费校验置信度下降，需在报告明确标注接入形态。
