## 1. 项目背景与目标

本项目是一个**中转站（API 代理）模型质量检测平台**（来源：项目 `CLAUDE.md`）。

**核心需求**
- 支持多协议中转站：OpenAI、Anthropic、**Gemini（Google Generative AI）** 三种上游协议，单个中转站可同时支持任意协议组合（如"兼容站"可同时勾选 OpenAI+Anthropic+Gemini）。
- Gemini 支持两种暴露形态：**原生协议**（`generativelanguage.googleapis.com`，`:generateContent`，`x-goog-api-key` 鉴权）与中转站用 **OpenAI 兼容层**（`/v1/chat/completions`，`model=gemini-*`）转出。
- 中转站可自定义**请求地址、API Key、名称**。
- 自动从中转站拉取支持的模型列表；拉取失败时支持手动输入。
- 输出各中转站不同模型的**质量报告**，可导出 **PDF**。
- **可视化**配置中转站与发起质量检测。

**核心价值定位（用户确认）**
> 平台最核心的价值是**模型来源真实性检测**——识别中转站是否"套壳换底"或"从 Codex 等订阅工具逆向转出 API 冒充官方直供"。其余维度（连通性、计费、能力）为辅助。

## 2. 需求澄清结论（决策记录）

| # | 决策点 | 选择 | 关键理由 |
|---|---|---|---|
| 1 | 部署形态 | **本地 Web 应用**（浏览器界面 + 本地存储 + Key 仅本地） | 部署简单、契合可视化需求、Key 不出本机更安全 |
| 2 | 检测维度 | 连通性与性能 / 计费一致性 / 能力探测 / **真实性检测（核心）** | 移除"输出正确性（题集评测）"，省去最大工作量与 token 消耗（YAGNI） |
| 3 | 真实性对照基线 | **无官方 key，纯启发式检测** | 无官方直连作对照，只能靠行为指纹/特征加权给"可疑度" |
| 4 | 真实性子目标 | a. 套壳/换底  b. **逆向/工具转出**（Codex / ChatGPT / Claude 订阅 / Cursor / Copilot / **Gemini CLI / AI Studio** 等逆向成 API） | 识别"非官方直供"来源 |
| 5 | 后端技术栈 | **Python（FastAPI 异步）** | tokenizer、异步探测、PDF 生态最成熟 |
| 6 | 前端技术栈 | **Vue 3 + Naive UI** | 配置型后台界面友好、中文社区活跃 |
| 7 | 数据存储 | **SQLite** | 本地轻量；遵循全局 DB 规则（自增主键/无外键/datetime/全注释） |
| 8 | **支持协议** | **OpenAI / Anthropic / Gemini 三协议** | 覆盖主流上游；适配器模式新增协议不改现有代码（SOLID-O） |
| 9 | **协议建模** | **协议集合多选**（站点存 `protocols` 集合，模型归属单一 `protocol`） | 取代固定的三/四类枚举组合，避免枚举组合爆炸，未来加协议零改枚举 |
| 10 | **Gemini 接入形态** | **原生协议（Developer 风格 + Vertex 风格两种端点）+ OpenAI 兼容层** 均支持 | 覆盖 AI Studio 直供、Vertex 原生路径透传、generativelanguage 风格封装、兼容层转出全部现状 |
| 11 | **Gemini 真实性主手段** | **功能性指纹探针：主动测能否跑通 Gemini 独有功能以鉴别"是不是真 Gemini"** | 真 Gemini 必能返回独有功能的特有结构化字段，套壳模型给不出；功能可用性比被动指纹更难伪造。上游 Studio/Vertex 仅附带信息，非判定重点 |

**术语约定**
- **TTFT（Time To First Token）**：流式请求中，从发出请求到收到第一个 token 的耗时，衡量"首字延迟"。非流式用"总耗时"。
- **套壳/换底**：中转站声称提供贵模型，实际转发给便宜/其他模型。
- **逆向/工具转出**：将 C 端订阅工具（Codex、ChatGPT Plus、Claude Pro、Cursor、Copilot、**Gemini CLI / Google AI Studio / Antigravity** 等）的额度逆向封装成兼容 API 出售，冒充官方直供。
- **协议（protocol）**：中转站上游遵循的 API 规范，取值 `openai`·`anthropic`·`gemini`。一个站可支持多协议；一个模型归属单一协议。
- **Gemini 原生协议**：Google Generative AI 的 REST 规范，端点 `:generateContent` / `:streamGenerateContent`，鉴权 `x-goog-api-key`，usage 字段为 `usageMetadata`。

## 3. 可行性分析

| 维度 | 可行性 | 实现路径 | 局限 |
|---|---|---|---|
| 连通性与性能 | ✅ 成熟 | HTTP 请求 + 计时 + 错误码统计 | 无 |
| 计费一致性 | ✅ 可行 | 申报单价 × usage + 本地 tokenizer 估算对比 + 官方基线对比（如有） | 看不到真实账单扣费；Anthropic 流式默认不返回 usage，需本地估算兜底 |
| 能力探测 | ✅ 可行 | 标准探针：上下文长度/流式/函数调用/多模态等 | 上下文长度探测较耗时（需递增输入） |
| **Gemini 协议适配** | ✅ 可行 | 原生 `:generateContent`/`:streamGenerateContent` + `usageMetadata`；`:countTokens` 精确计数；`GET /v1beta/models` 拉取模型；兼容层走 `/v1/chat/completions` | 兼容层下 Gemini 特有字段（safetyRatings/thoughtsTokenCount）可能被中转站抹平，真实性特征减弱 |
| 真实性检测（套壳+逆向转出） | ⚠️ 启发式可行 | 探针提取行为特征 + 加权评分 → 可信度分级 | **无法 100% 确认**；中转站可刻意抹除特征；仅给"高度可疑/可能/正常"分级 |
| PDF 报告 | ✅ 可行 | 前端 ECharts 渲染图表 + 后端 weasyprint 由 HTML 模板生成 | 无 |
| 本地部署 | ✅ 可行 | FastAPI + Vue 静态资源 + SQLite | 无 |

**结论**：项目整体技术可行。唯一需在产品层面诚实传达的是——**真实性检测是"可信度评分"而非"铁证"**。

```
                           │ HTTP / SSE

│                └─────┬──────┘                             │

│                └────────────┘                             │
└──────────────────────────┬──────────────────────────────┘
                           │

                   │   SQLite        │  中转站/模型/任务/结果/探针记录
                   └────────────────┘
```

**分层职责**
- **前端**：配置管理、任务发起与实时进度（SSE）、报告可视化与 PDF 触发。
- **后端 API 层**：CRUD、任务调度、SSE 推送。
- **检测引擎**：异步任务池执行探针序列，落库 `probe_record`。
- **Provider 适配器**：三类协议（OpenAI / Anthropic / Gemini）+ 兼容层统一为同一调用接口（适配器模式）。Gemini 适配器内部再分"原生协议"与"OpenAI 兼容层转出"两条路径。
- **真实性评分**：聚合探针特征 → 加权评分。
- **PDF 生成**：weasyprint 由 HTML 模板渲染。
- **数据层**：SQLite。

## 5. 关键技术决策

| 决策点 | 方案 | 理由 |
|---|---|---|
| 检测执行 | 异步任务池 + SSE 实时进度 | 单模型多探针耗时数十秒~分钟，阻塞会 HTTP 超时 |
| Provider 适配 | 适配器模式，OpenAI/Anthropic/Gemini 三协议统一接口 | 符合 SOLID-O，新增协议不改现有代码 |
| Token 计数 | tiktoken（OpenAI 兼容）+ 近似估算兜底（Anthropic）+ **Gemini `:countTokens` 端点 / 估算兜底** | 校验 usage 真实性 |
| PDF 导出 | 前端 ECharts 渲染 + 后端 weasyprint 由 HTML 生成 | 图表丰富、样式可控 |
| Key 存储 | 本地加密存储 + 日志脱敏 | 遵循安全规则（Key 不外发、不打印） |
| 数据层 | SQLite（自增主键/无外键/datetime/全字段注释） | 遵循全局 DB 规则 |
| 交付 | 开发式启动为主，可选 docker-compose（届时确认时区） | 本地自用最简 |

## 6. 数据模型设计

遵循全局规则 3.2：整数自增主键、无外键约束（业务字段关联）、`datetime` 类型、全字段带 COMMENT。

**① `relay_station`（中转站）**：id / name / protocols（**协议集合**，JSON 文本，元素 ∈ `openai`·`anthropic`·`gemini`；取代原单值 `station_type`，"兼容站"即多元素集合） / base_url / api_key(加密) / status / created_at / updated_at

**② `model`（模型）**：id / station_id(业务关联) / protocol（**该模型归属协议**，`openai`·`anthropic`·`gemini`，须 ∈ 所属站的 protocols） / access_mode（**接入形态**，`native` 原生协议·`openai_compat` 兼容层转出；非 Gemini 默认 `native`） / gemini_endpoint_style（**Gemini 原生端点形态**，`gemini_developer`·`vertex`，默认 `gemini_developer`；仅 gemini+native 有意义） / gemini_vertex_json（**Vertex 风格专属配置**，JSON 文本：`{project, location, auth_style}`；非 Vertex 留空，下放至模型级以支持同站不同模型属不同 project/location） / model_name / display_name / source(`fetched`·`manual`) / input_price(**TEXT 精确小数**，应用层 Decimal 计算) / output_price(**TEXT 精确小数**) / declared_context_length / enabled / created_at / updated_at

**③ `detection_task`（检测任务）**：id / station_id / model_id / status(`pending`·`running`·`completed`·`failed`·`canceled`) / progress / config_json(检测配置快照) / started_at / finished_at / created_at / updated_at / error_message

**④ `detection_result`（结果汇总）**：id / task_id / model_id / overall_score / connectivity_score / performance_score / billing_score / capability_score / authenticity_score / authenticity_subscores_json(套壳分+逆向转出分+逐条信号明细) / details_json(各维度评分计算明细与聚合上下文快照；真实性信号专置于 authenticity_subscores_json) / created_at

**⑤ `strategy_result`（策略检测结果）**：id / task_id / model_id(业务关联) / strategy_category（策略大类：`connectivity`·`performance`·`billing`·`capability`·`authenticity`） / strategy_key（策略细项标识，如 `ttft`·`throughput`·`billing_consistency`·`cap_function_call`·`gemini_thinking`·`gemini_code_execution`·`gemini_url_context`·`gemini_vertex_rag` 等） / strategy_name(策略中文名) / result_status（`pass`·`fail`·`degraded`·`skipped`；功能探针 supported/degraded/unsupported 映射于此） / score(该策略得分，可空) / weight(该策略在所属维度的权重快照) / metrics_json(量化指标，如 ttft_ms/命中的特有字段名) / evidence_json(判定证据，脱敏：命中/缺失字段、原始片段引用) / created_at
> **每个模型 × 每个检测策略一行**，独立保留每项策略的结论与证据，支撑可溯源的真伪/质量报告。

**⑥ `probe_record`（探针原始记录）**：id / task_id / strategy_result_id(业务关联，归属哪条策略结果) / probe_type / request_at / response_at / ttft_ms / http_status / success / usage_json / estimated_tokens / feature_flags_json / raw_response_json(脱敏) / error_message / created_at

**设计要点**
- 探针原始记录与结果汇总分离：支持溯源审计，评分逻辑可独立演进。
- 价格/上下文等可变信息快照进 `config_json`，保证历史报告可复现。
- **协议集合多选**：`relay_station.protocols` 存协议集合而非固定类型枚举，新增协议（未来如 Cohere/Bedrock）无需改枚举，仅扩集合取值与适配器（SOLID-O）。SQLite 无原生数组，用 JSON 文本字段承载，应用层解析。
- **模型归属单一协议**：每个 `model` 通过 `protocol` 明确归属，`access_mode` 区分 Gemini 走原生协议还是兼容层，决定探针调用哪条适配路径与如何解析 usage。
- **三层结果模型**：`detection_result`(模型综合分) → `strategy_result`(每模型每策略结论，逐项保留) → `probe_record`(策略下每次请求原始记录)，自顶向下可逐层下钻溯源。
- **价格精确存储**：SQLite 无原生 DECIMAL（NUMERIC 亲和性下小数退化为浮点），故 `input_price`/`output_price` 用 TEXT 存精确小数，应用层 Decimal 计算；若改用 PostgreSQL 则可直接用 NUMERIC。
- **Gemini 端点配置下放模型级**：`gemini_endpoint_style` 与 `gemini_vertex_json` 置于 `model`，支持同一中转站下不同 Gemini 模型属不同 project/location 与端点形态。
- **字段注释落地（SQLite 适配）**：全局 DB 规则 3.2 要求"全字段 COMMENT"，但 SQLite 不支持列级 COMMENT 语法；落地方式为建表 DDL 内以行内注释标注每列含义 + 同步维护 `doc/` 数据字典文档；若迁移至 PostgreSQL/MySQL 则改用原生 `COMMENT`。

## 7. Gemini 协议适配要点

Gemini（Google Generative AI）的 API 规范与 OpenAI/Anthropic 差异较大，且中转站存在两种暴露形态，适配器内部据此分两条路径。

### 7.1 两条接入路径

| 路径 | `access_mode` / `endpoint_style` | 端点与鉴权 | usage 字段 | 适用场景 |
|---|---|---|---|---|
| Gemini 原生 · Developer 风格 | `native` / `gemini_developer` | `POST {base}/v1beta/models/{model}:generateContent`；流式 `:streamGenerateContent?alt=sse`；鉴权 `x-goog-api-key: <KEY>`（或 `?key=`） | `usageMetadata.{promptTokenCount, candidatesTokenCount, totalTokenCount, thoughtsTokenCount}` | AI Studio 直供站、以 generativelanguage 风格封装上游（含封装后的 Vertex）的中转站 |
| Gemini 原生 · Vertex 风格 | `native` / `vertex` | `POST {base}/v1/projects/{project}/locations/{location}/publishers/google/models/{model}:generateContent`；流式 `:streamGenerateContent`；鉴权 `Authorization: Bearer <token>`（或站点自定义） | 同上 `usageMetadata` | 中转站以 Vertex 原生路径透传（需配 project/location） |
| OpenAI 兼容层转出 | `openai_compat` / — | `POST {base}/v1/chat/completions`，`model="gemini-*"` | OpenAI 风格 `usage.{prompt_tokens, completion_tokens, total_tokens}` | 多数综合中转站把 Gemini 并入兼容层 |

### 7.2 请求/响应结构差异（原生路径）

- **请求体**：`{"contents":[{"role":"user","parts":[{"text":"..."}]}], "generationConfig":{...}, "systemInstruction":{...}}`；角色仅 `user`/`model`，无 `system` role（用 `systemInstruction`）。
- **流式**：`:streamGenerateContent?alt=sse` 返回 SSE，每帧为完整 `GenerateContentResponse` 增量；TTFT 取首个含 `candidates[].content.parts[].text` 的帧。
- **能力探测**：函数调用走 `tools[].functionDeclarations`；多模态走 `parts[].inlineData`/`fileData`；JSON 模式走 `generationConfig.responseMimeType="application/json"`；思考模型（gemini-2.5 系）含 `thinkingConfig` 与 `thoughtsTokenCount`。
- **安全字段**：响应含 `candidates[].safetyRatings`、`promptFeedback.blockReason`，为 Gemini 特有指纹。

### 7.3 模型列表与 token 计数

- **模型列表**：Developer 风格 `GET {base}/v1beta/models`（返回 `models[].name`、`supportedGenerationMethods`、`inputTokenLimit`/`outputTokenLimit`）；Vertex 风格 `GET {base}/v1/projects/{project}/locations/{location}/publishers/google/models`；兼容层走 `GET {base}/v1/models`。拉取失败回退手动输入（与现有规则一致）。
- **token 计数**：Developer 风格 `POST {base}/v1beta/models/{model}:countTokens`、Vertex 风格 `.../models/{model}:countTokens` 可精确计数，用于校验 `usageMetadata` 真实性；兼容层无对应端点，用估算兜底。

### 7.4 真实性检测 — Gemini 逆向/套壳特征

延续"逆向/工具转出"子目标，Gemini 的非官方直供来源指纹：

| 来源 | 特征指纹 | 加权方向 |
|---|---|---|
| **Google AI Studio 逆向** | 免费档模型可用但缺企业级配额特征；`safetyRatings` 结构异常或被裁剪；限流模式贴近 AI Studio 免费额度 | 可疑↑ |
| **Gemini CLI / Code Assist 逆向** | 注入特定 system 指令/工具壳痕迹；`modelVersion` 回显缺失或与官方不符；响应头缺 Google 直供特征 | 可疑↑ |
| **Antigravity / IDE 工具转出** | agentic 工具特有的会话/上下文包装残留；多模态/函数调用能力被阉割 | 可疑↑ |
| **套壳换底（声称 Gemini 实转他模型）** | `usageMetadata` 缺失或字段不全（无 `thoughtsTokenCount` 但声称 2.5 思考模型）；不支持 Gemini 特有 `responseMimeType`/`safetyRatings`；tokenizer 切分特征与 Gemini 不符 | 高度可疑↑ |
| **官方直供（正常基线）** | `usageMetadata` 字段齐全且与 `:countTokens` 一致；`safetyRatings`/`modelVersion` 完整规范；能力探针全通过 | 正常 |

> 同其余真实性维度：**仅给可信度分级（高度可疑/可能/正常），非铁证**；阈值可调，支持人工复核。兼容层转出会抹平部分原生特征，此时真实性判定置信度自动下调并在报告标注。

### 7.5 功能性指纹探针（主动探测，Gemini 真实性核心手段）

**核心思想**：**真正的 Gemini 模型必然能正确使用 Gemini 独有功能**，而套壳成其他模型（GPT/Claude/开源模型）的中转站给不出这些功能特有的结构化响应。因此主动调用 Studio / Vertex 的独有功能，校验响应是否返回该功能**特有的结构化字段**——**能正确返回即证明是真 Gemini**；报错 / 忽略参数返回普通文本 / 特有字段缺失 = 并非真 Gemini（套壳换底）。功能可用性是判定"是不是真 Gemini"的核心试金石，远比被动指纹可靠。

> **本节目标是真伪鉴别（是不是真 Gemini），不是上游溯源**。能跑通 Gemini 功能的就是真 Gemini，至于功能由 Studio 还是 Vertex 提供，仅作附带信息，不作为判定重点。

**判定三态**：每个探针输出 `supported`（返回特有字段且语义正确）/ `degraded`（部分返回或字段不全）/ `unsupported`（报错或完全忽略）。

#### A. 通用 Gemini 独有功能探针（真伪主判据，双向加权）

两个平台的真 Gemini 都应支持（视模型版本），套壳成非 Gemini 模型则给不出特有字段。**支持→证真、声称支持却不支持→证伪**。

| 探针 | 请求要点 | 真 Gemini 特有字段（须出现） | 套壳模型表现 |
|---|---|---|---|
| **思考（Thinking）** | 2.5/3 思考模型设 `thinkingConfig`（`thinking_level` 或 `thinking_budget`） | `usageMetadata.thoughtsTokenCount>0`；多轮回传 `thoughtSignature` | 声称思考模型但 thoughtsTokenCount 缺失/为 0；无签名 |
| **代码执行** | `tools:[{code_execution:{}}]` + 要求"执行代码得结果" | `parts[].executableCode.{language,code}` + `parts[].codeExecutionResult.{outcome,output}` | 只回代码文本、无执行结果字段 |
| **Google 搜索接地** | `tools:[{google_search:{}}]`（旧模型 `google_search_retrieval`）+ 问需实时信息的问题 | `groundingMetadata.{webSearchQueries, groundingChunks[].web.{uri,title}, groundingSupports, searchEntryPoint}` | 无 groundingMetadata / 报错 / 纯文本臆造 |
| **受控结构化输出** | `generationConfig.responseMimeType="application/json"` + `responseSchema`（含 enum / propertyOrdering） | 严格遵循 schema，字段顺序符合 propertyOrdering | 忽略 schema / 返回非结构化文本 |
| **上下文缓存** | `cachedContents.create` 建缓存后带 `cachedContent` 引用请求 | `usageMetadata.cachedContentTokenCount>0` + 计费折扣 | 缓存端点不存在 / 字段缺失 |
| **logprobs** | `generationConfig.responseLogprobs=true` + `logprobs=N` | `candidates[].logprobsResult.{topCandidates, chosenCandidates}` + `avgLogprobs` | 字段缺失 |
| **安全评级** | 普通请求观察 | `candidates[].safetyRatings[].{category,probability}`、`promptFeedback.blockReason` 结构规范 | 缺失 / 结构与 Gemini 不符 |
| **token 计数一致性** | `:countTokens` 与实际 `usageMetadata.promptTokenCount` 对比 | 两者一致（Gemini 自有 tokenizer 切分） | 偏差大 / 端点不存在 |
| **多模态时间戳** | 传视频 `fileData` + 问"第 N 秒发生什么" | 准确定位时间戳内容 | 不支持多模态 / 答非所问 |

#### B. 平台特有功能探针（单向确证，支持=铁证真 Gemini，不支持不扣分）

仅某一平台提供，受接入方式限制。**支持即一票确证为真 Gemini；不支持不能据此判套壳**（可能只是另一平台直供或模型版本不含该功能）。

| 探针 | 平台 | 仅此平台支持的特征 | 用途 |
|---|---|---|---|
| **URL Context** | 仅 Developer API（Studio） | `url_context_metadata.urlMetadata[].{retrievedUrl, urlRetrievalStatus}` 且内容反映该页 | 通过即证真 Gemini |
| **Vertex RAG 检索接地** | 仅 Vertex | 接受 `tools:[{retrieval:{vertexRagStore:{...}}}]` 并返回检索结果 | 通过即证真 Gemini |
| **Google Maps 接地** | 仅 Vertex（Gemini 3 企业） | 返回 Maps 接地元数据 | 通过即证真 Gemini |
| **SafetySetting.method** | 仅 Vertex | 接受 `safetySettings[].method=PROBABILITY\|SEVERITY` 并生效 | 通过即证真 Gemini |

#### C. 真伪判定逻辑（是不是真 Gemini）

- **A 组任一探针 supported** → 该功能特有结构化字段非 Gemini 模型无法伪造，**强力证真**；多个 supported → 基本确认真 Gemini。
- **B 组任一探针 supported** → **一票确证真 Gemini**（平台特有功能别的模型绝无法返回对应结构）。
- **声称为 Gemini 模型，但 A 组功能（在该模型版本应支持的范围内）大面积 unsupported** → **疑似套壳成非 Gemini 模型**，真实性可疑度↑。
- **A、B 全部 unsupported 且仅基础对话可用** → 高度可疑（套壳，或仅透传基础 generateContent 的残缺逆向）。
- 上游来自 Studio 还是 Vertex 由 B 组附带得出，**仅作报告备注，不影响真伪结论**。

#### D. 探针执行约束

- **协议路径限制**：功能性探针**仅对 `access_mode=native` 原生路径有效**；OpenAI 兼容层（`/v1/chat/completions`）通常不透传这些 Gemini 专属工具，兼容层下偏门功能 unsupported 是**预期而非证据**，此时降级为弱信号并在报告标注"兼容层无法做功能性指纹检测"。
- **成本与开关**：偏门功能探针（尤其搜索接地、代码执行、视频）消耗较高，须可单独勾选、设预算上限、支持采样而非全跑（YAGNI / 风险节 token 消耗）。
- **误报控制**：单一探针 unsupported 不直接判套壳（可能模型版本不支持该功能），需结合模型声明能力（`supportedGenerationMethods`）与多探针综合加权。

## 8. 检测引擎与探针集详细设计

### 8.1 引擎执行模型

一次检测 = 一个 `detection_task`，针对单个（站点, 模型）。引擎按"探针序列"逐项执行，每个探针产出一条 `strategy_result` 与若干 `probe_record`。

**任务生命周期**：`pending → running → (completed | failed | canceled)`。

**执行流程**：
1. 加载任务配置快照（启用的探针集、并发数、重复次数、预算上限、阈值表）。
2. **连通性探针先行**：失败则短路（后续探针无意义），任务标注"不可用"后结束。
3. 其余探针按类别串行、类别内受控并发执行（受 `max_concurrency` 与 token 预算约束）。
4. 每个探针完成即落库 + 经 SSE 推送进度（`strategy_key / status / progress%`）。
5. 全部完成 → 评分聚合 → 写 `detection_result`。

**取消**：通过 cancel token 在每个探针边界检查；运行中请求设单次超时，任务级总超时兜底。

**进度计算**：`progress = 已完成探针数 / 启用探针总数`（连通性短路时直接置满）。

**并发层级**：两级并发控制——① **任务内**：同类别探针受 `max_concurrency` 与 token 预算约束并发；② **任务间**：引擎维护全局任务池（上限可配，本地场景建议 2~3），超出则排队，避免多任务并发耗尽中转站额度或集中触发限流。

### 8.2 探针统一抽象（适配器模式 + 策略模式）

```
Probe（抽象接口）
  key: str                       # 对应 strategy_result.strategy_key
  category: str                  # connectivity/performance/billing/capability/authenticity
  name: str                      # 策略中文名
  weight: float                  # 在所属维度内的权重
  applicable(model, ctx) -> bool # 该模型/接入形态是否适用（如功能性探针仅 native）
  async run(ctx) -> ProbeResult
```

- **ProbeContext**：持有 `ProviderAdapter`（已按协议/接入形态选定）、模型配置、预算计数器、cancel token、阈值表。探针不直接拼协议细节，统一经适配器收发，符合 SOLID-D（依赖抽象而非具体协议）。
- **ProbeResult → 落库映射**：
  - `status` ∈ `pass / degraded / fail / skipped`（功能探针的 supported/degraded/unsupported 映射至 pass/degraded/fail）
  - `score`（可空）、`metrics`（量化指标）、`evidence`（脱敏证据：命中/缺失字段名、原始片段引用）
  - 一条 ProbeResult 写一行 `strategy_result`；探针内每次 HTTP 往返写一行 `probe_record`（关联 `strategy_result_id`）。
- 新增探针只需实现接口并注册，不改引擎（SOLID-O）。

### 8.3 连通性与性能探针

| 探针 key | 判定逻辑 | metrics | 三态 |
|---|---|---|---|
| `connectivity` | 最小合法请求（1 token 输出），校验 HTTP 200 + 响应体结构合法 | http_status, 是否可解析 | pass=可用 / fail=不可达或鉴权失败 |
| `ttft` | 流式请求，记录首个有效内容帧耗时 | ttft_ms（多次取中位） | 阈值分级 pass/degraded |
| `throughput` | 流式请求，统计稳定段 token/s | tokens_per_sec, 总耗时 | 阈值分级 |
| `stability` | 重复 N 次，统计成功率与延迟方差 | success_rate, p95_ms | 阈值分级 |

- 重复次数、并发由配置控制；非流式模型 TTFT 退化为"总耗时"并标注。

### 8.4 计费一致性探针（key: `billing_consistency`）

- 发起固定提示词请求，取响应 usage（OpenAI `usage` / Anthropic `usage` / Gemini `usageMetadata`）。
- 本地 tokenizer 估算（tiktoken / Gemini `:countTokens` / Anthropic 近似）作对照。
- 比对项：申报 token 数 vs 本地估算偏差率；`申报单价 × usage` 的成本一致性（价格取 `model.input_price/output_price` 的精确小数，应用层 Decimal 计算）。
- **判定**：偏差 ≤ 阈值 → pass；偏差大或 usage 缺失 → degraded/fail，并作为真实性"套壳"弱信号联动。
- Anthropic 流式无 usage 时走本地估算兜底并降低该项置信度。

### 8.5 能力探测探针

| key | 探测内容 | 判定 |
|---|---|---|
| `cap_context_length` | 递增输入逼近申报上下文上限（设步长与硬上限，避免高消耗） | 实测可用上下文 vs 申报值 |
| `cap_streaming` | 是否支持 SSE 流式 | pass/fail |
| `cap_function_call` | 工具/函数调用是否返回结构化调用 | pass/degraded/fail |
| `cap_multimodal` | 图像/音视频输入是否被接受并理解 | pass/fail/skipped |
| `cap_json_mode` | 受控 JSON 输出是否生效 | pass/degraded |

- `applicable` 依据模型声明能力（如 `supportedGenerationMethods`）裁剪；不适用则 `skipped`，不计负分（避免误伤）。

### 8.6 真实性探针（套壳换底 + 逆向/工具转出）

落实第 7.4 节特征。每个特征项是一个探针或探针内的特征提取器，产出"可疑度信号"：
- **套壳换底信号**：usage 字段缺失/不全、协议特有字段（`safetyRatings`/`system_fingerprint` 等）异常、tokenizer 切分特征不符、能力探针大面积失败但声称高能力模型。
- **逆向/工具转出信号**：注入的 system/工具壳痕迹、`modelVersion`/`system_fingerprint` 缺失或与官方不符、限流模式贴近 C 端订阅额度、响应头缺官方直供特征。
- 每信号输出脱敏 `evidence` + 权重，汇入真实性评分（第 9 节，待展开）。

### 8.7 Gemini 功能性指纹探针（落实 7.5，仅 `access_mode=native`）

- **A 组（双向判真伪）**：thinking / 代码执行 / 搜索接地 / 受控结构化输出 / 上下文缓存 / logprobs / 安全评级 / token 计数一致性 / 多模态时间戳。每探针校验"特有结构化字段"是否出现并语义正确 → supported(pass) 强力证真；声称支持却 unsupported → 证伪信号。
- **B 组（单向确证）**：URL Context / Vertex RAG / Google Maps 接地 / SafetySetting.method。supported → 一票确证真 Gemini；unsupported 不扣分（`skipped` 或弱信号）。
- **执行约束**：兼容层路径下整组 `applicable=false`（标注"兼容层无法做功能性指纹检测"）；高成本探针（搜索 / 代码执行 / 视频）须可单独勾选 + 预算上限 + 采样执行；单探针 unsupported 不独立定性，须多探针加权。

### 8.8 结果落库与评分聚合

- 探针执行层只产出 `strategy_result` + `probe_record`，**不做最终评分**（评分逻辑独立演进，关注点分离）。
- 评分聚合层读取本任务全部 `strategy_result`，按维度权重计算 `connectivity_score / performance_score / billing_score / capability_score / authenticity_score`，合成 `overall_score`，写 `detection_result`。
- 真实性子分（套壳分 / 逆向转出分）单列入 `authenticity_subscores_json`；详细加权与分级阈值见第 9 节。

## 9. 真实性评分模型详细设计

> 本节是平台**核心价值**的落地。原则：**无罪推定 + 信号加权 + 一票确证/否决 + 置信度调节**，输出"可信度分级"而非铁证（第 3 节、9.7 节再次重申）。

### 9.1 评分目标与输出语义

- **两个独立子分**（各 0–100，越高越可信）：
  - `shell_score`（**非套壳可信度**）：模型是否真为其声称的型号（反"套壳换底"）。
  - `direct_score`（**官方直供可信度**）：是否官方直供而非 C 端工具逆向转出（反"逆向/工具转出"）。
- **合成** `authenticity_score = min(shell_score, direct_score)`：取短板——任一子目标高度可疑都应拉低总体可信度（而非平均稀释）。
- **三级分级**（阈值可调）：`正常 (≥H) / 可能可疑 (L~H) / 高度可疑 (<L)`，默认 `H=75 / L=45`。
- 每个子分附 `confidence`（0–1）：样本不足、兼容层抹平特征时下调，报告随分级一并展示。

### 9.2 信号模型（Signal）

每个真实性探针 / 特征提取器产出一个或多个 Signal：

```
Signal
  key        # 特征项标识，对应 strategy_result.strategy_key
  target     # shell | direct（作用于哪个子分）
  direction  # confirm(证真) | refute(证伪)
  severity   # 0~1，信号强度（命中程度，三态映射：hit=1 / degraded=0.5 / miss=0）
  weight     # 该特征基础权重
  confidence # 0~1，本信号可信度（受接入形态/样本量影响）
  evidence   # 脱敏证据（命中/缺失字段、原始片段引用）
```

**子分计算**（无罪推定，从满分扣减）：

```
base = 100
refute 信号：score -= Σ (weight × severity × confidence)        # 可疑→扣分
confirm 信号：score = min(100, score + Σ(weight × severity × confidence × 回补系数))
            # 证真回补，但不超过 100；强确证另见 9.5 一票确证
score = clamp(score, 0, 100)
```

### 9.3 套壳换底评分（`shell_score`，落实 7.4 + 8.6）

| 特征项 key | 方向 | 基础权重 | 命中含义 |
|---|---|---|---|
| `shell_usage_missing` | refute | 30 | usage/`usageMetadata` 缺失或字段不全（声称思考模型却无 `thoughtsTokenCount`） |
| `shell_special_field_absent` | refute | 25 | 协议特有字段缺失（`safetyRatings`/`system_fingerprint`/`logprobs` 结构不符） |
| `shell_tokenizer_mismatch` | refute | 20 | 本地 tokenizer 切分与申报 usage 偏差显著（联动 8.4 计费探针） |
| `shell_capability_gap` | refute | 15 | 声称高能力模型但能力探针大面积 fail（函数调用/多模态/JSON 模式） |
| `shell_feature_confirm` | confirm | — | Gemini A 组功能 supported（见 9.5，强证真该模型确为 Gemini） |

### 9.4 逆向/工具转出评分（`direct_score`，落实 7.4 + 8.6）

| 特征项 key | 方向 | 基础权重 | 命中含义 |
|---|---|---|---|
| `reverse_shell_artifact` | refute | 25 | 注入的 system/工具壳痕迹（Codex/CLI/IDE 工具的会话包装残留） |
| `reverse_version_anomaly` | refute | 20 | `modelVersion`/`system_fingerprint` 缺失或与官方不符 |
| `reverse_ratelimit_pattern` | refute | 20 | 限流模式贴近 C 端订阅额度而非 API 配额 |
| `reverse_header_missing` | refute | 15 | 响应头缺官方直供特征 |
| `reverse_studio_signature` | refute | 15 | Gemini：免费档配额特征 / `safetyRatings` 被裁剪（贴近 AI Studio 逆向） |

### 9.5 Gemini 功能性指纹融入（落实 7.5 → 评分）

- **A 组 supported** → `shell_feature_confirm` 强证真：单个 supported 即把 `shell_score` 下限抬升（套壳成非 Gemini 模型无法返回特有结构化字段）；多个 supported → `shell_score` 基本封顶可信。
- **A 组"声称支持却 unsupported"** → 转为 `shell_*` refute 信号（证伪），严重度依模型版本应支持程度加权。
- **B 组 supported** → **一票确证**：`shell_score` 直接锁定高可信（≥H），并在 `direct_score` 标注"平台特有功能可用，官方直供旁证"。
- **B 组 unsupported** → 不扣分（`skipped`/弱信号），避免误伤 Developer API 直供或不含该功能的版本。
- **兼容层路径（`access_mode=openai_compat`）** → 功能性探针整组不可用，相关信号 `confidence` 置低并在报告标注"兼容层无法做功能性指纹检测，真实性判定置信度下降"。

### 9.6 置信度（confidence）调节规则

`authenticity_score` 的展示分级须结合 confidence；以下因素下调 confidence：
1. **兼容层抹平**：经 OpenAI 兼容层转出，原生指纹丢失 → confidence × 0.6。
2. **样本不足**：性能/计费类重复次数低、功能探针采样未覆盖 → 按覆盖率折算。
3. **单信号定性**：仅凭单一 refute 信号不下"高度可疑"结论，需多信号或强信号支撑（误报控制）。
4. **模型版本不匹配**：某功能本就不属该模型版本能力范围 → 对应 unsupported 不计 refute。

### 9.7 可解释性与人工复核

- 评分聚合时将每个 Signal 的 `(key, direction, severity, weight, confidence, 贡献分值, evidence)` 写入 `detection_result.authenticity_subscores_json`，逐条可溯源（关联 `strategy_result` / `probe_record`）。
- 报告按"贡献分值"降序展示 Top 信号，标明证真/证伪与脱敏证据，支持人工复核与阈值调整。
- **产品措辞红线**：始终呈现"可信度分级"，不承诺"100% 识别"；阈值 `H/L`、各特征 `weight`、`回补系数` 全部可配置，便于按实际样本校准。

## 10. 核心数据流与 API 设计

### 10.1 端到端检测流程

```
配置中转站 → 拉取/录入模型 → 发起检测任务 → 引擎异步执行探针序列
   → SSE 实时进度 → 策略结果/原始记录落库 → 评分聚合 → 报告/PDF 导出
```

时序：
1. `POST /api/stations` 创建中转站（协议集合、base_url、api_key 加密入库）。
2. `POST /api/stations/{id}/models/fetch` 自动拉取模型；失败回退 `POST /api/stations/{id}/models` 手动录入。
3. `POST /api/tasks`（station_id, model_id, config）→ 返回 `task_id`，引擎异步启动。
4. `GET /api/tasks/{id}/events`（SSE）订阅进度。
5. 引擎逐探针落 `strategy_result`/`probe_record` → 评分聚合写 `detection_result`。
6. `GET /api/tasks/{id}/result` 取综合结果；`GET /api/tasks/{id}/report.pdf` 导出。

### 10.2 REST 接口契约

| 分组 | Method & Path | 说明 |
|---|---|---|
| 中转站 | `POST /api/stations` | 创建（protocols 集合 / base_url / api_key / name） |
| | `GET /api/stations` · `GET /api/stations/{id}` | 列表 / 详情（**响应不含 api_key 明文**，仅返回是否已配置） |
| | `PUT /api/stations/{id}` · `DELETE /api/stations/{id}` | 更新 / 删除 |
| | `POST /api/stations/{id}/models/fetch` | 拉取上游模型列表，失败返回可回退手输标志 |
| 模型 | `GET /api/stations/{id}/models` | 该站模型列表 |
| | `POST /api/stations/{id}/models` | 手动录入（source=manual，含 access_mode/gemini 端点配置/价格） |
| | `PUT /api/models/{id}` · `DELETE /api/models/{id}` | 更新 / 删除 |
| 检测任务 | `POST /api/tasks` | 创建并异步启动，返回 task_id |

### 10.3 SSE 进度事件契约

`GET /api/tasks/{id}/events` 推送以下事件（`event: <type>` + `data: <json>`）：

| event | data 关键字段 | 时机 |
|---|---|---|
| `task.started` | `task_id, total_probes` | 任务开始 |
| `probe.completed` | `strategy_key, status, progress, metrics(脱敏)` | 每个探针完成 |
| `task.scored` | `overall_score, authenticity_score` | 评分聚合完成 |
| `task.completed` | `task_id, result_id` | 全部结束 |
| `task.failed` | `task_id, error_message(脱敏)` | 异常终止 |

> **连通性短路**时直接发 `task.completed` 并在结果标注"不可用"。

### 10.5 安全与约束（落实安全规则 3.5）

- **api_key 永不外发**：任何响应 / SSE / 报告 / 日志均**不回显** key 明文（station 详情仅返回 `has_api_key: true`）；key 仅服务端→上游请求时使用。
- **不自动登录、不改上游数据**：检测仅发探测性推理请求，绝不触发上游侧写操作。
- **写操作校验**：创建任务前校验 model 归属 station、protocol ∈ station.protocols、Gemini Vertex 配置完整性。
- **预算护栏**：`budget_tokens` 超限时停止后续高成本探针并在报告标注"因预算未跑完整探针集"。
- **api_key 加密主密钥管理**：api_key 以对称加密落库，主密钥（master key）优先取自环境变量；缺省时首次启动生成并存于本地受限权限文件（不随库提交）。主密钥本身绝不入库、不入日志、不回显（落实安全规则 3.5）。

### 10.6 模型拉取编排（多协议站）

`POST /api/stations/{id}/models/fetch` 需处理单个站点同时支持多协议（`protocols` 集合）的情形，编排规则如下：

1. **逐协议并行拉取**：遍历 `station.protocols`，对每个协议调用其对应模型列表端点：
   - `openai` → `GET {base}/v1/models`
   - `anthropic` → `GET {base}/v1/models`（Anthropic 模型列表端点）
   - `gemini` → 原生 Developer 风格 `GET {base}/v1beta/models`；Vertex 风格 `GET {base}/v1/projects/{project}/locations/{location}/publishers/google/models`（见 7.3）
2. **protocol 归属推导**：每个返回模型按其来源端点标注 `protocol`，天然满足 `protocol ∈ station.protocols` 约束（10.5 校验）。
3. **access_mode 推导**：
   - 从 OpenAI 兼容层 `/v1/models` 拉到的 `gemini-*` 模型 → `access_mode=openai_compat`；
   - 从 Gemini 原生 `v1beta/models` 或 Vertex 端点拉到的 → `access_mode=native`（并据来源端点置 `gemini_endpoint_style`）；
   - 非 Gemini 协议模型 → `access_mode=native`（默认）。
4. **合并去重**：以 `(protocol, model_name, access_mode)` 为唯一键合并多协议结果；同名模型在不同协议/接入形态下视为不同条目，各自独立检测。
5. **部分失败处理**：某协议端点拉取失败不阻断其他协议；返回**逐协议拉取状态**（成功/失败 + 失败原因脱敏），失败协议回退手动录入（`source=manual`，与现有规则一致）。
6. **响应结构**：`{ fetched: [{protocol, access_mode, model_name, ...}], failures: [{protocol, reason}], fallback_manual: bool }`。

> 设计落点：fetch 不做检测、不落 `detection_*`，仅写/更新 `model` 表；多协议聚合让"兼容站"一次拉取即覆盖全部上游模型。

## 11. 错误处理与边界设计

> 检测过程频繁与不可控的上游交互，错误是常态而非例外。本节定义**统一错误分类、重试退避、探针级容错、边界条件**，确保单点失败不污染整体结论，且全程不泄露 Key。

### 11.1 错误分类体系

所有异常归一为统一错误类别，按"是否可重试 / 是否致命短路"决定处置：

| 错误类别 | 典型触发 | HTTP 特征 | 可重试 | 探针处置 | 是否短路全任务 |
|---|---|---|---|---|---|
| `connectivity_error` | DNS 失败 / 连接拒绝 / TLS 异常 | 无响应 | 有限重试 | fail | **是**（连通性探针失败即短路） |
| `auth_error` | Key 无效 / 过期 / 权限不足 | 401 / 403 | 否 | fail | **是**（标注"Key 失效"） |
| `quota_exceeded` | 额度耗尽 / 欠费 | 402 / 403 / 特定文案 | 否 | fail | **是**（标注"额度不足"） |
| `rate_limit` | 触发限流 | 429 | **是**（退避） | 重试后判定 | 否 |
| `timeout` | 单请求超时 / 任务总超时 | 无响应 | 有限重试 | degraded/fail | 否（任务级超时除外） |
| `upstream_5xx` | 上游网关/服务异常 | 500/502/503/504 | **是**（退避） | 重试后判定 | 否 |
| `parse_error` | 响应结构非法 / JSON 解析失败 / 缺必备字段 | 200 但体异常 | 否 | fail | 否 |
| `capability_unsupported` | 模型不支持该功能/参数 | 400 参数错误 | 否 | **skipped**（不计负分） | 否 |
| `budget_exceeded` | 本地 token 预算护栏触发 | 本地拦截 | 否 | 后续高成本探针 skipped | 否（任务正常结束并标注） |

### 11.2 统一错误模型与脱敏

```
ProbeError
  category      # 上表九类之一
  retryable     # bool
  http_status   # int | None
  message       # 面向用户的简述（已脱敏）
  raw_excerpt   # 原始响应片段引用（脱敏：剥离任何 key/authorization 回显）
```

- **脱敏强制**（落实安全规则 3.5）：错误 `message`/`raw_excerpt`/日志中**一律不含 Key、token、Authorization 头**；若上游错误体回显了凭据，落库前先正则剥离。
- `ProbeError` 写入 `probe_record.error_message`（脱敏）与 `strategy_result.evidence_json`，供报告与溯源使用。

### 11.3 重试与退避策略

- **可重试类**（`rate_limit` / `timeout` / `upstream_5xx`）：**指数退避 + 随机抖动**，最大重试次数可配（默认 2），优先尊重 `Retry-After` 响应头。
- **不可重试类**（`auth` / `quota` / `parse` / `capability`）：立即失败或短路，不浪费额度。
- **重试核算**：重试消耗计入 token 预算与耗时统计；重试请求**不参与 TTFT 中位计算**（仅取成功的稳定样本），并在 metrics 标注重试次数。

### 11.4 探针级容错（单点失败隔离）

- **隔离原则**：单个探针抛错被引擎捕获，写 `strategy_result(status=fail/skipped)` 后**继续执行后续探针**，不终止整个任务（关注点分离 + 韧性）。
- **致命短路例外**：仅 `connectivity_error` / `auth_error` / `quota_exceeded` 触发全任务短路（后续探针无意义）。
- **部分成功降级**：重复 N 次的探针（如 `stability`/`ttft`）部分失败时，按**成功率降级**（degraded）而非整体判 fail，方差/成功率写入 metrics。

### 11.5 边界条件处理

| 边界场景 | 处理策略 |
|---|---|
| **usage 字段缺失**（Anthropic 流式默认无 usage） | 本地 tokenizer 估算兜底；计费一致性 `confidence` 下调；同时作为套壳弱信号联动（9.3 `shell_usage_missing` 视严重度判定） |
| **流式中途断连** | 已收 token 计入；标注 `incomplete`；`stability` 记一次失败；不据残缺响应做能力/真实性定性 |
| **正常截断 vs 异常截断** | 区分 `finish_reason=length/stop`（正常）与 `safety/error`（异常）；正常截断不计失败 |
| **能力不支持**（400 参数错误） | 对应 `cap_*`/功能性探针置 `skipped`，**不计负分**（避免误伤版本差异） |
| **上下文长度探测触顶** | 输入过大被拒时记录"实测可用上限"，正常结束探针，不判错误 |
| **功能性探针 @ 兼容层** | `applicable=false` 跳过，标注"兼容层无法做功能性指纹检测"（落实 7.5-D / 9.5） |
| **Gemini Vertex 配置缺失** | 任务创建前校验 `project/location/auth_style` 完整性，缺失则拒绝创建（**不发出无效请求**，落实 10.5 写操作校验） |
| **空响应 / 模型返回空内容** | 判 `parse_error` 或 degraded，依语义区分；不静默当成功 |

### 11.6 任务级超时与取消边界

- **双层超时**：单请求超时（连接超时 / 读取超时分离配置）+ **任务总超时兜底**（防个别探针卡死拖垮整任务）。
- **取消语义**：`cancel token` 在每个探针边界检查；运行中请求收到取消后尽力中止，**已落库的 `strategy_result`/`probe_record` 保留**。
- **终态**：超时→`failed`，主动取消→`canceled`；`error_message`（脱敏）写入 `detection_task`，并经 SSE 发 `task.failed`。

### 11.7 错误对评分与报告的影响

- **致命短路任务**：`overall_score` 标注"不可用"，仅呈现连通性结论，其余维度标注"未检测"，不强行给分。
- **三态对评分的差异处置**：`skipped` 不计入分母（不扣分）；`degraded` 按比例计权；`fail` 计为该策略权重缺失。
- **检测完整性区块**：报告专设"检测完整性"段，明示哪些探针**未跑 / 失败 / 降级**及脱敏原因（如"因预算未跑完整探针集""兼容层跳过功能性指纹"），与真实性 `confidence` 联动展示，避免用户误读残缺结论。

## 12. 报告与 PDF 设计

> 报告是检测价值的最终交付物。设计原则：**真实性专章优先呈现（核心价值）、每项结论可下钻溯源、红线措辞贯穿全篇、敏感信息全程脱敏**。

### 12.1 报告结构

报告按"概览→核心→辅助维度→完整性→附录"自上而下组织，对应数据来源见 12.4：

| 序 | 区块 | 内容要点 | 主数据来源 |
|---|---|---|---|
| 1 | **报告头** | 站点名 / 模型显示名 / 协议 + 接入形态 / 检测时间 / 任务 ID（**不含 api_key**） | `relay_station`·`model`·`detection_task` |
| 2 | **综合评分概览** | `overall_score` + 五维（连通/性能/计费/能力/真实性）雷达图 + 分级标签 | `detection_result` |
| 3 | **真实性专章（核心）** | 分级徽标（正常/可能可疑/高度可疑）+ `shell_score`/`direct_score` 双子分 + `confidence` + Top 信号贡献条形图 + 脱敏证据表 + **红线声明** | `detection_result.authenticity_subscores_json` |
| 4 | **连通性与性能** | TTFT 中位 / 吞吐 / 稳定性（成功率·p95），折线/柱状 | `strategy_result`(connectivity/performance) |
| 5 | **计费一致性** | 申报 token vs 本地估算偏差、成本核算（Decimal） | `strategy_result`(billing) + `probe_record.usage_json` |
| 6 | **能力探测矩阵** | 各能力项 pass/degraded/fail/skipped 状态格 | `strategy_result`(capability) |
| 7 | **Gemini 功能性指纹**（仅 native 且适用时） | A 组双向 / B 组确证探针结果表 + 证真/证伪标注 | `strategy_result`(authenticity, gemini_*) |
| 8 | **检测完整性** | 未跑/失败/降级探针清单及脱敏原因（联动 11.7） | `strategy_result.result_status` 汇总 |
| 9 | **附录·逐策略明细** | 每条 `strategy_result` 的 metrics/evidence，可下钻 `probe_record` | 全量 `strategy_result` |

### 12.2 可视化图表（ECharts）

| 图表 | 类型 | 承载区块 |
|---|---|---|
| 五维评分雷达 | radar | 综合概览 |
| 真实性信号贡献 | 横向 bar（证真绿/证伪红，按贡献绝对值降序） | 真实性专章 |
| TTFT / 吞吐分布 | line / boxplot（多次重复样本） | 性能 |
| 计费偏差对比 | 分组 bar（申报 vs 估算） | 计费 |
| 能力矩阵 | 状态色块表（pass/degraded/fail/skipped 四色） | 能力 |

- 图表配色与三态/分级语义统一（绿=正常/证真、橙=可疑/降级、红=高度可疑/失败、灰=跳过）。

### 12.3 PDF 导出流程

**技术选型**（落实第 5 节决策：前端 ECharts 渲染 + 后端 weasyprint）：为**避免后端引入无头浏览器**，采用"前端渲染图表为静态图 → 后端 HTML 模板合成"流程：

```
前端报告页渲染 ECharts → 各图 chart.getDataURL() 得 PNG(base64)
   → POST /api/tasks/{id}/report.pdf  body: { charts: { radar: <b64>, ... } }
   → 后端取 detection_result/strategy_result 组装数据
   → Jinja2 HTML 模板内联图片 + 数据表
   → weasyprint 渲染 HTML→PDF → 返回二进制流（Content-Disposition: attachment）
```

- **接口修订**（衔接 10.2）：报告导出由 `GET` 调整为 **`POST /api/tasks/{id}/report.pdf`**（需接收前端渲染的图表 base64）；GET 形式仅返回纯数据表版 PDF（无图，作为无前端图表时的降级）。
- **模板与样式**：HTML/CSS 模板（A4、分页、页眉页脚含任务 ID 与生成时间），中文字体内嵌确保 weasyprint 正确渲染。
- **脱敏强制**：模板渲染前对所有字段过滤 api_key/token；`evidence` 仅含字段名与脱敏片段引用。

### 12.4 报告数据来源映射

- 报告**只读** `detection_result`（综合分与真实性子分 JSON）+ `strategy_result`（逐策略结论）+ 按需下钻 `probe_record`（脱敏原始记录），不重新发起检测请求。
- 历史可复现：报告所用价格/上下文/阈值均取自 `detection_task.config_json` 快照，保证旧报告与当时配置一致。

### 12.5 红线措辞与脱敏

- **真实性结论红线**：报告全程使用"可信度分级 / 旁证 / 疑似"措辞，**不出现"100% 确认/铁证"**（落实第 3 节、9.7）；分级旁强制展示 `confidence` 与"启发式检测，建议人工复核"提示。
- **接入形态标注**：兼容层转出的模型在真实性专章显著标注"兼容层无法做功能性指纹检测，置信度下降"。
- **敏感信息**：报告、PDF、文件名、日志一律不含 Key/token；站点仅显示名称与 `has_api_key`。

## 13. 测试策略

> 检测平台自身的正确性直接决定结论可信度。测试核心目标：**判定逻辑可复现、不依赖真实中转站、绝不外发 Key**。所有上游交互以录制夹具（fixture）打桩，单元测试零网络。

### 13.1 测试分层与目标

| 层级 | 对象 | 目标 | 是否触网 |
|---|---|---|---|
| 单元 | 探针判定 / 评分函数 / 适配器解析 | 纯逻辑正确性，三态与分值精确 | 否（夹具打桩） |
| 集成 | 引擎调度 + 适配器 + 落库 | 任务生命周期、并发、短路、SSE 事件序列 | 否（mock 传输层） |
| 端到端 | API → 引擎 → 评分 → 报告 | 关键用户流贯通（创建任务→结果→PDF） | 否/可选录制回放 |

### 13.2 探针单元测试

- **三态判定**：对每个探针喂入"特有字段齐全 / 部分缺失 / 完全缺失"三类夹具，断言 `pass/degraded/fail/skipped` 与 `metrics`、`evidence` 字段正确。
- **applicable 裁剪**：验证功能性探针在 `access_mode=openai_compat` 时 `applicable=false`（不执行不计分）。
- **重点用例**：计费一致性偏差阈值边界；能力探测 `skipped` 不计负分；连通性失败触发短路标志。

### 13.3 评分模型测试（核心）

落实第 9 节，按"信号 → 子分 → 合成 → 分级"逐层验证：

- **信号加权**：构造 refute/confirm 信号集，断言 `shell_score`/`direct_score` 扣分与回补计算（含 `severity` 三态映射、`confidence` 系数）。
- **一票确证/否决**：B 组 supported → `shell_score` 锁定 ≥H；声称支持却 unsupported → 转 refute 拉低分。
- **取短板合成**：`authenticity_score = min(shell, direct)`，验证单维度高度可疑不被另一维度平均稀释。
- **分级与置信度**：跨 H/L 阈值的边界值；兼容层 `confidence × 0.6`；样本不足按覆盖率折算；单信号不下"高度可疑"。
- **阈值可配**：调整 `H/L`、`weight`、回补系数后分级随之变化，验证无硬编码。

### 13.4 适配器测试（OpenAI / Anthropic / Gemini 双路径）

- **请求构造**：断言各协议请求体、鉴权头正确（Gemini Developer `x-goog-api-key`、Vertex `Authorization: Bearer`、兼容层 `/v1/chat/completions`）；**断言请求与日志中均不含明文 Key 落盘**。
- **响应解析**：用录制夹具验证 usage 字段映射（OpenAI `usage` / Anthropic `usage` / Gemini `usageMetadata` 含 `thoughtsTokenCount`）、流式 TTFT 帧解析、Gemini 特有结构（`groundingMetadata`/`safetyRatings`/`executableCode`）提取。
- **双路径用例**：同一 Gemini 模型分别走 `gemini_developer`、`vertex`、`openai_compat` 三套夹具，验证端点拼装与 usage 解析分支正确。

### 13.5 错误处理与边界测试（落实第 11 节）

- **错误分类映射**：401→`auth_error` 短路、429→`rate_limit` 退避重试、5xx→重试、400 参数→`capability_unsupported` skipped。
- **重试退避**：mock 时钟断言指数退避 + 抖动 + 尊重 `Retry-After`；重试不污染 TTFT 中位。
- **边界**：Anthropic 流式无 usage 走估算兜底并降置信度；流式中断标 incomplete；正常截断 vs 异常截断区分；Vertex 配置缺失任务创建前被拒（不发请求）。

### 13.6 端到端流程测试

- **任务生命周期**：`pending→running→completed/failed/canceled` 状态流转；连通性短路直达 completed 并标"不可用"。
- **SSE 事件序列**：断言 `task.started → probe.completed* → task.scored → task.completed` 顺序与脱敏字段。
- **落库与报告**：三层结果（result/strategy/probe）写入正确；报告数据只读快照、PDF 导出（POST 带图表 base64）生成成功且**输出中无 Key**。

### 13.7 测试数据与打桩

- **录制夹具**：以脱敏的真实响应样本（各协议、各探针的成功/降级/失败/异常变体）作为 fixture，单元/集成测试零网络、可复现。
- **传输层 mock**：在 HTTP 客户端层注入 mock，断言请求构造同时返回预置响应，不连任何真实中转站。
- **安全约束**：夹具与测试代码**严禁包含真实 Key**；断言脱敏逻辑剥离 Key/token（安全规则 3.5 的回归保护）。

### 13.8 覆盖重点与执行

- **覆盖优先级**：评分模型 > 真实性探针判定 > 适配器解析 > 错误处理 > 引擎调度（按对结论可信度的影响排序）。
- **执行方式**：本地 `pytest`（后端）；前端组件/报告渲染按需测试。私有项目不使用付费 CI，构建测试在本地完成（落实规则 3.8）。

## 14. 风险与局限

1. **真实性检测非铁证**：启发式评分，中转站可对抗。报告须明示"可信度"语义，避免向用户承诺"100% 识别"。
2. **Anthropic usage 兜底**：流式默认不返回 usage，计费一致性需本地估算，精度受限。
3. **上下文长度探测耗时**：递增输入策略需设上限，避免长时间占用与高额消耗。
4. **token 消耗**：检测本身消耗中转站额度，需提供并发数/重复次数/预算上限配置。
5. **逆向探针误报**：行为特征可能误判，阈值需可调并支持人工复核。
6. **Gemini 兼容层特征丢失**：经 OpenAI 兼容层转出的 Gemini 会抹平 `safetyRatings`/`usageMetadata`/`thoughtsTokenCount` 等原生指纹，真实性与计费校验置信度下降，需在报告明确标注接入形态。
